# Evaluation：怎么判断链路是真的变好了

这一节对应 `07-evaluation.py`。

很多人做 LLM 应用时，能把流程跑起来，却很难回答一个更关键的问题：

> 我改了模型、改了提示词、改了检索策略以后，效果到底是变好了还是变坏了？

这一节就是在解决这个问题。

## 1. 为什么评估比“看起来还行”更重要

LLM 应用有个非常常见的错觉：

- 你试了两个样例，感觉回答不错
- 于是你以为系统整体不错

但一旦输入分布变化、用户问题变复杂，效果就可能迅速下滑。

所以评估的意义在于：

1. 给系统建立基线
2. 让修改前后可以对比
3. 把“主观感觉”变成“可重复检查”

## 2. 这个脚本是怎么组织的

`07-evaluation.py` 沿用了上一节的问答系统，然后增加了两层能力：

1. 构造测试样本
2. 给模型回答打分

文件里同时展示了：

- 人工编写样例
- LLM 自动生成样例
- LLM 评估回答是否正确

这很符合今天很多 AI 应用的实践方式。

## 3. 先有被评估对象：问答链

代码先复用了上一节的 RAG 主体：

```python
qa = RetrievalQA.from_chain_type(
    llm=model,
    chain_type="stuff",
    retriever=index.vectorstore.as_retriever(),
    verbose=True,
)
```

这一步非常重要，因为评估从来不是“空对空”的。你先得有一个稳定的被测系统。

## 4. 第一种样例来源：人工构造

```python
examples = [
    {
        "query": "舒适居家套头套装有侧边口袋吗？",
        "answer": "是"
    },
    ...
]
```

这是最直接的评估集构造方式。

优点：

- 你最清楚业务里什么问题重要
- 样例质量更容易把控

缺点：

- 人工成本高
- 扩展慢

## 5. 第二种样例来源：让 LLM 帮你出题

这一节很有代表性，因为它展示了“用 LLM 评估 LLM 应用”的经典思路。

### 自定义生成提示词

```python
QA_GENERATE_PROMPT = PromptTemplate(...)
```

这个 prompt 的目的不是回答问题，而是：

- 给一段文档
- 生成一组基于文档的问题和标准答案

### 自定义解析器

```python
QA_GENERATE_PARSER = RegexParser(...)
```

这里项目没有直接让模型返回 JSON，而是用正则把“问题：... / 答案：...”这种格式抽出来。

这说明评估链路里，**解析稳定性一样重要**。

### 生成样例

```python
example_gen_chain = QAGenerateChain(
    llm=model,
    prompt=QA_GENERATE_PROMPT,
    output_parser=QA_GENERATE_PARSER,
)
```

再通过：

```python
example_gen_chain.apply([{"doc": t} for t in data[:5]])
```

批量生成更多测试题。

## 6. 第三步：真的去跑模型回答

```python
predictions = qa.apply(examples)
```

这里的意思很直接：

- 拿评估集里的 query
- 交给问答链回答
- 收集系统输出

现在你手里就同时有了：

- 问题
- 标准答案
- 模型答案

这时才进入真正评分环节。

## 7. 第四步：让 LLM 当裁判

```python
eval_chain = QAEvalChain.from_llm(model, prompt=QA_EVAL_PROMPT)
graded_outputs = eval_chain.evaluate(examples, predictions)
```

这是本节最核心的一步。

项目自定义了中文评估提示词，但最终评分标签仍保持：

- `CORRECT`
- `INCORRECT`

这样做的好处是：

- 更贴近当前中文数据环境
- 同时保持框架容易解析的固定标签

## 8. 为什么这套做法有现实意义

因为很多 LLM 系统的问题，不是传统单元测试能完全覆盖的。

例如：

- 回答是否忠于检索文档
- 总结是否遗漏关键信息
- 语言是否与需求一致

这些任务天然更适合：

- 样例集
- 评分标准
- LLM judge

## 9. 但你也要知道它的边界

LLM 评估不是绝对真理，它也会有偏差。

常见问题包括：

- 裁判模型自己判断不稳
- 评分 prompt 写得不够清晰
- 标准答案太短，导致“部分正确”难判断

所以正确心态应该是：

> LLM 评估能大幅提升效率，但最好和人工 spot check 结合使用。

## 10. 当前项目相对视频的学习价值

### 价值 1：把评估 prompt 中文化了

这让你更容易看懂“评估到底在判什么”。

### 价值 2：展示了输出解析在评估链中的作用

很多人只在业务生成里想到 parser，其实评估链更依赖 parser，因为样例集构造和评分结果都需要更稳定的结构化输出。

### 价值 3：把 debug 能力也串进来了

```python
set_debug(True)
```

这提醒你一件事：评估不仅是最后打分，也包括过程观察。

## 11. 真实项目里怎么落地这一节

一个比较实用的迭代闭环是：

1. 先做一小批人工黄金样例
2. 再用 LLM 自动扩样
3. 每次改 prompt、改模型、改检索策略后重跑
4. 比较通过率变化
5. 对异常样本做人工复盘

这样你才会真正拥有“可演进的 AI 应用”，而不是一堆靠感觉调出来的 prompt。

## 12. 建议你自己动手改的练习

1. 给 `examples` 再手工补 5 个问题，覆盖价格、材质、功能等不同维度。
2. 修改 `QA_EVAL_PROMPT`，让评估标准更严格，比如要求答案必须明确引用商品属性。
3. 比较“只用人工样例”和“人工 + 自动生成样例”时的评估结果差异。
